초안

ClassMate 운영 가이드

이 문서는 프로그램 사용 방법만 설명하는 문서가 아닙니다. 실제 업무에서 필요한 입력 조건, 판정 규칙, 예외 처리 기준, 운영 시 주의사항까지 함께 정리한 사내 운영 문서입니다.

0. 이 문서의 범위

이 문서는 세 가지 관점으로 구성합니다.

사용자 가이드

실제 작업자가 프로그램을 사용하는 순서대로 설명합니다. 실행 전 준비사항, 스캔, 실행, 결과 확인, 명단 반영, 명단비교 모드를 포함합니다.

관리자 가이드

안내문, 템플릿, resources 구성처럼 운영 자산을 유지하거나 교체할 때 필요한 내용을 설명합니다.

개발·유지보수 참고사항

프로그램 구조, 주요 파일 역할, 규칙 수정 시 영향 범위를 설명합니다. 함수명 자체를 나열하기보다 수정 포인트를 찾을 수 있는 수준으로 정리합니다.

작성 원칙설계 내용은 숨기지 않습니다. 다만 함수명이나 내부 변수명 대신, 실제 업무에서 이해 가능한 표현으로 바꿔 설명합니다.

1. 사용자 가이드

이 섹션은 실제 프로그램 화면과 같은 순서로 읽을 수 있도록 구성합니다. 사용자는 이 섹션만 읽어도 프로그램을 실행하고, 오류 원인을 확인하고, 결과를 해석할 수 있어야 합니다.

1-1. 시작 전 준비사항

작업 폴더와 resources 폴더

이 프로그램은 작업 폴더 안의 구조를 기준으로 파일을 찾습니다. 따라서 작업 시작 전에 폴더 구성이 먼저 맞아 있어야 합니다.

항목필수 조건확인 방법

templates 폴더등록 템플릿 1개, 안내 템플릿 1개가 있어야 합니다.파일명에 각각 등록, 안내 포함 여부 확인

DB 폴더학교전체명단 xlsb 파일이 있어야 합니다.전체 명단 조회와 학교 검색에 사용
예시resources 폴더가 2개 있거나, templates 안에 등록 템플릿이 2개 있으면 프로그램은 어떤 파일을 써야 할지 결정할 수 없으므로 바로 오류를 표시합니다.

학교 폴더 안 입력 파일

학교 폴더 안의 파일명은 프로그램이 자동으로 종류를 구분하는 기준입니다. 파일 내용이 맞더라도 이름이 맞지 않으면 프로그램이 찾지 못합니다.

입력 파일 헤더 규칙

프로그램은 헤더 행을 자동으로 찾고, 그 헤더 이름을 기준으로 학년·반·이름 같은 필수 열을 읽습니다. 따라서 학교에서 받은 파일의 열 제목이 너무 다르면 스캔에서 막힐 수 있습니다.

기본 원칙신입생 / 전입생 / 전출생 / 재학생 비교 파일은 최소한 학년, 반, 이름을 읽을 수 있어야 합니다. 교직원 파일은 이름과 신청 열을 읽을 수 있어야 합니다.
예시학년 열이 3학년, 반 열이 2반처럼 적혀 있으면 대부분 경고로 처리됩니다. 반대로 학년 열과 반 열이 하나로 합쳐져 3-2처럼 들어 있으면 구조 경고가 더 강하게 뜰 수 있습니다.

학생명부가 필요한 경우

  • 전입생 파일이 있는 경우
  • 전출생 파일이 있는 경우
  • 신입생 파일 안에 1학년 외 학년이 섞여 있어 아이디 시작 연도를 자동으로 추정해야 하는 경우

이 경우 학생명부가 없으면 일부 작업은 실행 자체가 되지 않거나, 보류가 발생합니다.

1-4. 스캔 검수

스캔은 입력 파일 구조를 읽고, 실행 가능한 상태인지 먼저 점검하는 단계입니다. 이 단계에서 파일 종류, 시트, 시작행, 구조 경고를 확인합니다.

시작행 확인

프로그램이 자동으로 헤더를 찾지 못하면 시작행이 어긋날 수 있습니다. 이 경우 미리보기에서 실제 헤더 행과 데이터 시작 위치를 확인한 뒤, 시작행 값을 조정합니다.

확인 체크

확인 체크는 해당 파일을 검토했다는 표시입니다. 시작행을 조정한 경우에는 조정 후 내용을 다시 확인한 뒤 체크합니다.

오류와 경고

오류가 있으면 실행할 수 없습니다. 경고는 실행은 가능하지만, 결과를 반드시 다시 확인해야 한다는 뜻입니다.

예시전입생 파일에서 헤더를 찾을 수 없습니다.가 뜨면 전입생 파일이 잘못된 것이 아니라, 헤더 행 자동 감지가 실패했을 가능성이 큽니다. 먼저 시작행과 미리보기를 함께 확인합니다.

실행 메모

실행 메모는 이번 작업의 상황을 남기는 용도입니다. 예를 들어 학교에서 재학생 명단을 다시 보낸 경우, 학생명부 기준일을 별도로 확인한 경우, 수동 보정이 필요한 사항이 있는 경우 간단히 적어 둡니다.

출력 파일 확인

  • 출력 파일명 클릭: 결과 파일 바로 열기
  • 폴더 열기: 작업 폴더의 결과 위치 열기
  • 동명이인만 보기: 노란색 표시가 있는 대상만 빠르게 점검

1-6. 안내문

안내문 탭은 resources/notices 안의 txt 파일을 기준으로 목록이 표시됩니다. 학교명, 학년도, 개학일, 도메인 같은 값은 프로그램이 현재 작업 기준으로 반영합니다.

안내문 탭에서 확인할 것목록에 필요한 안내문이 보이는지, 현재 학교에 맞게 문안이 채워졌는지, 복사 후 바로 발송 가능한 수준인지 확인합니다.

1-7. 명단 반영

명단 반영은 전체 명단 파일의 해당 학교 행에 작업 결과를 기록하는 단계입니다. 이 기능은 실행 완료 후 사용합니다.

기록되는 항목

신입생, 전입생, 전출생, 교직원 처리 결과와 작업자명이 기록됩니다. 도착일과 발송일은 체크된 경우에만 기록됩니다.

사용 순서

1) 메일 기록 날짜 확인 → 2) 도착일/발송일 체크 여부 확인 → 3) 명단 반영 클릭 → 4) 반영 완료 상태 확인

예시도착일 체크 없이 명단 반영을 누르면 처리 결과와 작업자명은 기록되지만, 도착일은 비워 둘 수 있습니다.

1-8. 명단비교 모드

명단비교 모드는 비교용 명단과 학생명부를 대조해, 이미 계정이 있는 학생과 추가 확인이 필요한 학생을 분류하는 모드입니다. 이 모드 자체가 등록을 수행하는 것은 아닙니다.

2. 관리자 가이드

이 섹션은 운영 자산을 유지하거나 교체할 때 참고합니다. 일반 작업자는 읽지 않아도 되지만, 템플릿이나 안내문 내용을 바꿔야 하는 경우에는 반드시 확인해야 합니다.

2-1. 안내문 txt 관리

resources/notices 폴더의 txt 파일명이 그대로 안내문 목록의 제목이 됩니다. 파일 추가, 삭제, 수정은 이 폴더에서 관리합니다.

2-2. 등록 템플릿 수정 범위

등록 템플릿은 시트명과 핵심 열 제목을 프로그램이 직접 참조합니다. 서식은 바꿔도 되지만, 시트명과 필수 열 제목은 유지해야 합니다.

2-3. 안내 템플릿 수정 범위

안내 템플릿은 4행부터 데이터가 들어갑니다. 1~3행은 자유롭게 수정할 수 있지만, 데이터 시작 위치는 유지하는 것이 안전합니다.

2-4. 템플릿 교체 시 점검

  • 파일명에 등록 또는 안내 키워드가 포함되어 있는지
  • 같은 종류 템플릿이 2개 이상 들어 있지 않은지
  • 교체 후 스캔에서 오류가 없는지

3. 규칙 설명

이 섹션은 프로그램이 실제로 어떤 기준으로 판단하는지 설명합니다. 사용자도 결과를 이해하기 위해 알아야 하는 내용입니다.

3-1. 동명이인 규칙

같은 학년 안에서 이름이 같은 학생이 둘 이상 있으면 이름 뒤에 A, B, C 같은 구분자를 붙여 구분합니다.

상황처리 기준예시
신입생 파일 안에서 이름이 같음A부터 순서대로 붙임이수현, 이수현 → 이수현A, 이수현B
명부에 기존 동명이인이 있고 전입생 추가명부에 있는 마지막 구분자 다음부터 붙임김지우, 김지우A → 전입생 김지우B
중복 없음구분자 없이 사용홍길동

동명이인으로 처리된 행은 등록작업파일에서 표시되며, 안내 파일에도 구분자가 붙은 이름 그대로 출력됩니다.

3-2. 전입 처리 규칙

전입생은 학생명부를 기준으로 아이디 시작 연도를 추정합니다. 해당 학년 학생명부 정보가 충분하면 자동으로 처리하고, 판단 근거가 부족하면 보류합니다.

예시3학년 전입생을 처리할 때, 학생명부의 현재 3학년 학생 아이디가 대부분 2022로 시작하면 프로그램은 3학년 전입생도 2022 기준으로 아이디를 생성합니다.

3-3. 전출 처리 규칙

전출생은 학생명부에서 현재 반 / 이전 반 / 이름 / 아이디를 기준으로 매칭합니다. 한 명으로 확정할 수 있으면 퇴원 목록에 넣고, 여러 후보가 걸리면 보류합니다.

3-4. 보류 / 자동 제외

보류

보류는 프로그램이 자동으로 처리하기 어렵다고 판단한 경우입니다. 담당자가 확인 후 다시 실행하거나 수동 처리해야 합니다.

  • 전입생이 신입생 명단과 학년·반·이름이 같은 경우
  • 전입생이 명부에 이미 존재하는 경우
  • 전출생이 명부의 동명이인 후보 여러 명과 겹치는 경우
  • 명단비교에서 전입/전출을 자동 판정하기 어려운 경우

자동 제외

자동 제외는 명부에서 찾을 수 없거나, 현재 처리 대상이 아니라고 판단된 경우입니다. 자동 제외된 항목도 결과 파일에 사유와 함께 기록됩니다.

3-5. 명부 기준일

명부 기준일은 학생명부 파일의 마지막 수정일을 의미합니다. 프로그램은 이 날짜를 보고 현재 명부가 올해 기준인지, 개학 전 기준인지 해석합니다.

예시작업일은 3월 10일인데 학생명부 수정일이 2월 20일이라면, 개학 전 명부일 수 있으므로 학년 보정이 필요한지 확인합니다.

3-6. 학년도 아이디 규칙

사이드바의 학년도 아이디 규칙은 각 학년의 아이디 시작 연도를 뜻합니다. 학생명부가 있으면 스캔 후 자동으로 채워지고, 자동으로 판단하기 어려운 학년은 직접 수정할 수 있습니다.

직접 수정이 필요한 경우학생명부에 해당 학년 정보가 없거나, 신입생 파일에 여러 학년이 섞여 있는데 학생명부가 없을 때

3-7. 명단비교 분류 기준

명단비교는 이름, 학년, 반 정보를 기준으로 명단과 명부를 대조합니다.

분류뜻후속 조치
공통 학생이미 계정이 있는 재학생추가 조치 없음
명단에만 있음신규 등록 필요 가능성반편성 모드 전입/신입 검토
명부에만 있음전출 가능성반편성 모드 전출 검토
판정 불가자동으로 결론을 낼 수 없음결과 파일 수동 확인

영역역할대표 파일

규칙 수정 시 기본 원칙
