### 메세지 아키텍쳐
변경사항
맞다. 지금처럼

로그 원문
카드 문구
토스트 문구
상단 상태문구
가 제각각이면 위계가 무너진다.

이건 기능 문제가 아니라 메시지 아키텍처 문제다.

원인
현재 구조가 아마 이렇게 섞여 있다.

코어가 원문 로그를 만든다
브릿지가 그대로 넘긴다
프론트가 상황별로 또 문구를 만든다
어떤 건 카드, 어떤 건 토스트, 어떤 건 상단 텍스트로 따로 표현한다

그래서 같은 사건인데도

길이가 다르고
톤이 다르고
보여주는 위치도 다르고
어떤 건 파일명 포함, 어떤 건 없음
이렇게 된다.

조치
이건 기준을 딱 정해야 한다. 이렇게 가면 된다.

1) 메시지 위계 고정

로그

상세 원문 유지
디버깅/추적용
예: [ERROR] 전입생 파일에서 헤더를 찾을 수 없습니다. 학년/반/이름/NO 열을 확인해 주세요.

카드

사용자 행동 유도용 요약
짧고 통일
예: 헤더를 찾을 수 없습니다. 파일 구조를 확인해 주세요.

토스트

카드와 같은 문구
휘발성 알림만 담당
새 문구 만들지 않음

상단 상태문구

진행 상태만 표시
오류/경고 상세 설명 금지
예: 오류가 있습니다. 스캔 검수를 확인해 주세요.
2) 메시지 소스 단일화

프론트 여기저기서 문자열 직접 쓰지 말고
표준 메시지 매핑 함수 하나로 뽑아야 한다.

예:

header_missing → 헤더를 찾을 수 없습니다. 파일 구조를 확인해 주세요.
multi_sheet → 시트가 여러 개 있습니다. 첫 번째 시트만 사용합니다.
roster_missing → 학생명부 파일이 없습니다. 파일을 추가해 주세요.
freshmen_extra_grade → 1학년 외 신입생이 있습니다. 명단을 확인해 주세요.

즉
코어는 코드/원문,
UI는 코드 기반 표준 문구
이렇게 나누는 게 맞다.

3) severity와 display를 분리

한 사건마다 최소 이 정도는 있어야 유지보수 된다.

code: header_missing
severity: error | warn | info
scope: scan | run
target: freshmen | transfer_in | transfer_out | staff
raw_message: 원문 로그
display_message: UI 표준 문구

이렇게 가면

행 색
뱃지
카드
토스트
를 전부 같은 기준으로 움직일 수 있다.
4) 사용자 노출 규칙 고정

예를 들면 이렇게.

error

빨간 뱃지
빨간 카드

warn

노란 뱃지
노란 카드
해당 행 노란색
실행 가능 여부는 케이스별

info

로그에만 남김
카드 없음

이렇게 규칙화해야
“왜 이건 카드 뜨고 저건 안 뜨냐”가 안 생긴다.

5) 지금 프로젝트에 바로 적용할 최소 정리 순서

severity별 스타일 규칙 고정
